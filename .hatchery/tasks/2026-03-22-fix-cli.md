## Objective

When installing this package and trying to use the CLI, you will get an error like this:

FileNotFoundError: [Errno 2] No such file or directory: '/Users/grado/Code/projects/gpt-league/.venv/lib/python3.12/vault.yml'

We need a way to let the user specify credentials, either via .env file, vault file (with password file), or environment variables.

## Context

`vault.py` computed `VAULT_PATH` as `ROOT.parent.parent / "vault.yml"` where `ROOT` is the installed package location. When running from source this resolves to the repo root, but when installed in a venv it resolves to `<venv>/lib/python3.12/` — wrong. The same issue affected `CACHE_PATH` and `SECRETS_PATH` in `__init__.py`.

## Summary

### Key decisions

- **CWD as the default root**: CLI users run from their project directory, so `Path.cwd()` is the right default for runtime files like `vault.yml`, `secrets.json`, `.cache`. This mirrors the old source-run behaviour without depending on the package install path.
- **Env var overrides**: `VAULT_PATH`, `VAULT_PASSWORD_FILE`, `CACHE_PATH`, `SECRETS_PATH` all accept env var overrides for flexibility.
- **Graceful degradation**: `get_vault()` returns `{}` (instead of crashing) when vault file or password file is absent. Credentials can come from `.env` or env vars without needing a vault at all.
- **`.env` support**: `load_dotenv()` is called in `get_vault()` so a `.env` in the project directory is picked up automatically.
- `ROOT`, `ELO_MMR_PATH`, `ELO_MMR_CACHE_PATH`, `ELO_MMR_RESULT_PATH` in `__init__.py` were left unchanged — they are dev/submodule paths not needed at runtime.

### Files changed

- `pyproject.toml` — added `python-dotenv>=1.0.0`
- `src/gpt_racing/vault.py` — full rewrite; removed broken `VAULT_PATH` constant
- `src/gpt_racing/__init__.py` — `CACHE_PATH` and `SECRETS_PATH` now use env vars / CWD

### Gotchas for future agents

- `vault.py` no longer imports `ROOT` from `gpt_racing` — the circular-ish dependency is gone.
- The `yaml` import remains unused in the rewritten `vault.py` (it was unused before too, carried over from the original). Safe to remove in a follow-up cleanup.
